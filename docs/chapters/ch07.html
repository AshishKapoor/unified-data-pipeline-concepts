Ch 07 — Routing Data: Partition, Side Inputs, Tagged Outputs

Real pipelines don't push every element down one straight path — they route. Three tools do the routing: deterministic N-way splits (Partition), broadcast reference data (side inputs), and the dead-letter pattern (tagged outputs).

Concept

Up to now a transform took one PCollection and produced one PCollection. This chapter is about the cases where that one-in/one-out shape isn't enough: you need to fan a stream out into several, enrich elements with a shared lookup table, or peel off the records that don't belong in the main flow.

1 · beam.Partition — deterministic N-way split

Partition takes one PCollection and a function, and returns a fixed list of N PCollections. For each element it calls fn(element, num_partitions) → int and routes the element to that index. The function must be pure and deterministic: the same element always lands in the same partition. It's a labelled fan-out switch — not a filter, not a shuffle. Use it to split by tier, by region, by an A/B bucket, by "needs reprocessing vs not".

2 · Side inputs — broadcast a small dataset to every worker

A side input lets a ParDo read a second, usually small PCollection as ordinary in-memory data, available to every parallel worker. Beam materialises it once and broadcasts it — no per-element shuffle, no join. You choose how the main DoFn sees the side PCollection with a view:

  • beam.pvalue.AsSingleton — exactly one value (e.g. a threshold).
  • beam.pvalue.AsList — the whole thing as a Python list.
  • beam.pvalue.AsDict — a dict from a PCollection of (k, v) pairs.
  • beam.pvalue.AsIter — a re-iterable view (avoids materialising a giant list).

The trade-off: side inputs are for small data. Broadcasting a large table to every worker will blow up memory — that's when you reach for a CoGroupByKey join (Ch 5) instead. Here our "known regions" table is a handful of rows, the perfect side input.

🎯
The one idea for this chapter: a single ParDo can emit to more than one PCollection. Tag the streams, and bad records stop being a crash — they become a queryable side output you can alert on and replay. That's the dead-letter pattern.

3 · Tagged outputs — the dead-letter pattern

Normally a DoFn yields elements to one output. With .with_outputs(...) you declare a set of tags, and inside process you steer each element with yield beam.pvalue.TaggedOutput(tag, element) (a bare yield still goes to the main output). The classic use: a validator sends clean records to the main output and malformed ones to a dead_letter tag — annotated with why they failed — so one bad record never kills the whole job and nothing is silently dropped.

Beam APIs introduced

  • beam.Partition(fn, n) — split one PCollection into n by a pure function.
  • beam.pvalue.AsDict / AsList / AsSingleton / AsIter — side-input views.
  • ParDo(...).with_outputs(tag, ..., main=...) — declare additional outputs.
  • beam.pvalue.TaggedOutput(tag, value) — route an element to a named output.
  • DoFnOutputs.tag attribute access (e.g. results.valid, results['dead_letter']).

The example pipeline

We validate synthetic orders against a broadcast lookup table, send the good ones to the main output and split them three ways by amount tier, and divert every malformed record (unknown region, non-positive amount, missing field) to a dead_letter tag: 

class ValidateOrder(beam.DoFn):
    def process(self, record, regions):          # `regions` arrives as a side input
        if "region" not in record or "amount" not in record:
            yield beam.pvalue.TaggedOutput("dead_letter", {**record, "_error": "missing field"})
            return
        if record["region"] not in regions:       # the broadcast AsDict lookup
            yield beam.pvalue.TaggedOutput("dead_letter", {**record, "_error": "unknown region"})
            return
        if record["amount"] <= 0:
            yield beam.pvalue.TaggedOutput("dead_letter", {**record, "_error": "bad amount"})
            return
        yield {**record, "region_name": regions[record["region"]]}   # -> main output

with beam.Pipeline(options=options) as p:
    regions_si = p | "RegionsTable" >> beam.Create(list(KNOWN_REGIONS.items()))
    orders     = p | "ReadOrders"   >> beam.Create(RAW_ORDERS)

    results = orders | "Validate" >> beam.ParDo(
        ValidateOrder(), beam.pvalue.AsDict(regions_si)         # broadcast the table
    ).with_outputs("dead_letter", main="valid")

    valid, dead = results.valid, results["dead_letter"]
    small, medium, large = valid | "ByAmountTier" >> beam.Partition(amount_tier, 3)
    dead | "WriteDead" >> beam.io.WriteToText(known.output + "-dead_letter")

Notice there is no join and no shuffle to validate: the table was broadcast once and is just a Python dict inside process. The valid stream then fans out via Partition, and the bad stream is persisted for inspection.
