Ch 04 — Running on Flink for Real: The Portable Runner Architecture

Python doesn't run inside Flink. The submitter ships a language-neutral pipeline proto, the Job Server turns it into a Flink JobGraph, and TaskManagers call back to a Python SDK harness over the Fn API to execute your code.

Concept: the portability boundary

Flink is a JVM engine; your transforms are Python. Those two facts cannot be reconciled by magic, and the portable runner is how Beam reconciles them honestly. When you submit, four distinct things happen, in order:

  1. Submitter builds a pipeline proto. Your Python constructs the pipeline in memory, then Beam serialises it into a language-neutral proto (the Runner API). Each DoFn becomes an opaque blob plus a description of the environment needed to run it. Nothing JVM-specific is required to read it.
  2. Job Server → JobGraph. The proto travels to the Beam Flink Job Server (--job_endpoint). The Job Server stages your artifacts and translates the proto into a Flink JobGraph — this translation is literally what "the runner" is.
  3. JobManager schedules it. The JobGraph is submitted to Flink's JobManager, which schedules tasks across TaskManager slots exactly like a native Flink job.
  4. TaskManager ⇄ SDK harness over the Fn API. When a TaskManager reaches an operator that wraps your Python, it can't run Python itself. It calls back to a Beam SDK harness over the Fn API — four gRPC channels: control (run this bundle), data (elements in/out), state (read/write user state & side inputs), and logging. The harness runs your DoFn and streams results back.
🎯
The one idea for this chapter: the runner is a translator, and the Fn API is the contract across the language boundary. Your Python never runs on a TaskManager — it runs in an SDK harness that the TaskManager talks to.

Where does the harness live? --environment_type

The proto says which environment each transform needs; --environment_type tells the runner how to provide it. This single flag is the biggest operational lever in portable Beam:

  • LOOPBACK — the harness runs inside the submitting process. Zero extra infrastructure, ideal for a laptop; but the submitter must stay alive for the whole job. This is the dev lane this chapter demonstrates.
  • DOCKER — the runner starts a per-worker SDK container (Docker-in-Docker). Self-contained but heavy and awkward inside a cluster; avoided in this project.
  • PROCESS — the runner forks the harness as a plain OS process from a command you supply. Lighter than Docker, no daemon needed.
  • EXTERNAL — the harness lives in a pre-started worker pool sidecar that shares the TaskManager's network namespace (reached at localhost:50000). This is this project's production lane: the pool is up before any job, so there is no per-bundle container start-up cost.
⚠️
Artifact staging is part of submission, not execution. Pip dependencies, your code, and pinned files are uploaded to the --artifact_endpoint before the JobGraph runs, then distributed to harnesses. If staging is misconfigured, the job fails before your first element is ever processed.

Beam APIs introduced

  • PortableRunner — the runner that speaks the portability framework to a Job Server.
  • --job_endpoint — address of the Beam Flink Job Server that builds the JobGraph.
  • --artifact_endpoint — where the submitter stages code/deps before execution.
  • --environment_type / --environment_config — where the SDK harness lives (LOOPBACK / EXTERNAL / PROCESS / DOCKER).
  • Fn API — the control / data / state / logging gRPC channels between TaskManager and harness.

The example pipeline

The graph is a deliberately tiny WordCount — the lesson is in the journey, not the counting. The pipeline logs its effective submission contract so you can see the runner / job_endpoint / environment_type your run actually used: 

std      = options.view_as(StandardOptions)
portable = options.view_as(PortableOptions)

logging.info("runner            = %s", std.runner)             # PortableRunner
logging.info("job_endpoint      = %s", portable.job_endpoint)         # -> Beam Flink Job Server
logging.info("artifact_endpoint = %s", portable.artifact_endpoint)   # -> artifact staging
logging.info("environment_type  = %s", portable.environment_type)    # LOOPBACK / EXTERNAL / ...

with beam.Pipeline(options=options) as p:
    counts = (
        p
        | "ReadLines"   >> beam.io.ReadFromText(known.input)
        | "Tokenize"    >> beam.FlatMap(split_words)     # runs in the SDK harness, not Flink
        | "PairWithOne" >> beam.Map(lambda w: (w, 1))
        | "CountPerWord">> combiners.Count.PerKey()       # the shuffle lands on Flink
        | "Format"      >> beam.MapTuple(lambda w, n: f"{w}: {n}")
    )
    counts | "WriteCounts" >> beam.io.WriteToText(known.output, file_name_suffix=".txt")

Watch the log: the submitter-side logging.info calls reach this panel over SSE, but any print inside split_words would land in the harness logging channel and show up in the TaskManager logs instead — a direct consequence of where your code actually executes.
