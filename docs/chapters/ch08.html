Ch 08 — The Streaming Mindset: WHAT / WHERE / WHEN / HOW

Every event has two clocks: when it happened (event time) and when you saw it (processing time). The gap is skew, and because of it, out-of-order arrival is the default. Four questions tame the chaos: WHAT / WHERE / WHEN / HOW.

Concept — two clocks, never the same

In batch you have the luxury of a complete, finished dataset: there is effectively only one clock, because everything is already here. Streaming removes that luxury. Now every element carries an event time — the instant the thing actually occurred, stamped at the source (a click, a sensor reading, a sale). That timestamp is intrinsic to the data and never changes.

Against it runs processing time — the wall-clock instant at which your pipeline observes the element. Processing time depends on the network, on queues and retries, on GC pauses and scheduling. So the two clocks drift, and the gap between them is skew.

🎯
The one idea for this chapter: because of skew, out-of-order arrival is the default, not the exception. An event that happened earlier can be observed later than one that happened after it. Any correct streaming design must assume the stream is unsorted.

A picture helps: plot each element with processing time on the x-axis and event time on the y-axis. If the two clocks agreed, every point would sit on the diagonal. In reality points fall below the diagonal — event time lags the wall clock — and they scatter, because skew is not constant. That is exactly what the animation on the right shows.

The four questions

Beam (the Dataflow model) structures every streaming computation as the answers to four questions. This chapter only makes the first one concrete; the next four chapters answer the rest in turn — but knowing the whole map up front is what makes them click.

  • WHAT results are computed? → transforms (Map, GroupByKey, Combine). This chapter.
  • WHERE in event time? → windowing (Ch 9: fixed, sliding, sessions, global).
  • WHEN in processing time are results emitted? → watermarks and triggers (Ch 10–11).
  • HOW do successive refinements relate? → accumulation mode (discarding vs accumulating, Ch 11).
🧭
A useful mantra: WHERE lives in event time, WHEN lives in processing time. You decide which events belong together by their event time (windowing), but you decide when to fire a result by watching processing time and the watermark.

Beam APIs introduced

  • beam.window.TimestampedValue — wraps a record with an explicit event-time timestamp (here applied for us by ReplayEvents).
  • beam.DoFn.TimestampParam — a magic process() parameter that injects the element's event time into your DoFn.
  • time.time() inside process() — reads processing time; the difference from the event time is the skew.
  • _common.replay.ReplayEvents — replays records in a chosen emission_order so arrival order deliberately differs from event-time order.

The example pipeline

Five events spaced 10 seconds apart in event time, but replayed out of order — so the pipeline sees event time jump backwards. A DoFn logs both clocks and their skew for every element: 

# arrival order != event-time order: C(+20) arrives before A(+0) ...
EMISSION_ORDER = [2, 0, 4, 1, 3]   # -> C, A, E, B, D

class LogSkew(beam.DoFn):
    def process(self, element, event_ts=beam.DoFn.TimestampParam):
        processing_now = time.time()                 # WHEN we observed it
        event_seconds  = float(event_ts)             # WHEN it happened
        skew = processing_now - event_seconds        # the gap
        logging.info("id=%s EVENT_TIME=%.0f PROCESSING_TIME=%.3f SKEW=%.3fs",
                     element["id"], event_seconds, processing_now, skew)
        yield {**element, "skew": skew}

with beam.Pipeline(options=options) as p:
    (
        p
        | "ReplayOutOfOrder"          >> ReplayEvents(EVENTS, emission_order=EMISSION_ORDER)
        | "LogEventVsProcessingTime"  >> beam.ParDo(LogSkew())
    )

There is no WindowInto, no trigger, no accumulation here yet — that is the whole point. First feel the two clocks; Chapters 9–12 then answer WHERE, WHEN, and HOW. Because the SKEW lines are emitted by a worker DoFn, they appear in the TaskManager logs in the Flink UI rather than the submitter's stdout.
