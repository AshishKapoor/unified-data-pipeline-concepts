Ch 14 — Splittable DoFn (SDF): The Modern IO Primitive

Generalize DoFn so that processing one element is a restriction of work, tracked by a restriction tracker that supports try_claim() and can be split into primary + residual at any moment. This one idea powers dynamic work rebalancing, mid-element checkpointing, and every modern IO connector.

Concept

A plain DoFn processes an element atomically: process() runs start to finish, and the only parallelism the runner can exploit is splitting work across elements. That breaks down for a source, where a single "element" can stand for a whole file, a Kafka partition, or an infinite stream. You cannot hand half of one fat element to another worker, and you cannot persist progress halfway through reading it.

A Splittable DoFn makes the unit of work finer than an element. The work for one element is described by a restriction — a portion of work, e.g. the offset range [0, N). A restriction tracker owns that restriction and gates progress through try_claim(position):

  • try_claim(pos) returns True → the position is yours; do the work (read/emit) and advance.
  • try_claim(pos) returns False → the runner has split the range; that position now belongs to someone else. Stop immediately and return.

At any time the runner may call try_split(fraction) on the tracker, cleaving the remaining work into a primary (this worker keeps and finishes it) and a residual (returned to the runner to schedule elsewhere). That single move is what the diagram on the right animates.

🎯
The one idea for this chapter: an SDF turns "process this element" into "claim positions within a splittable restriction." Because the restriction can be split and checkpointed mid-element, the runner gets dynamic work rebalancing and durable progress for free — which is precisely why IO connectors are built on SDF.

Why this matters: three superpowers

  • Dynamic work rebalancing. A straggler element's unclaimed tail is peeled off to an idle worker mid-flight — no waiting for one slow bundle to drain.
  • Checkpointing mid-element. A try_split at fraction 0.0 checkpoints the residual durably, so an unbounded read can persist progress without ever "finishing" the element.
  • Bounded and unbounded restrictions. [0, N) is bounded; "tail this Kafka partition forever" is the unbounded restriction [offset, +∞).
⚙️
How IO connectors use this: ReadFromText, ReadFromKafka, and PubSub readers are all SDFs. Files split by byte offset; Kafka splits by partition and offset; both claim positions and report a watermark. You meet KafkaIO and the cross-language expansion service in Ch 15.
🧪
Advanced (named, not used here): unbounded restrictions pair with a watermark estimator (WatermarkEstimatorProvider, ManualWatermarkEstimator) so the SDF can advance event time as it reads. Custom watermark SDFs are out of scope for this lab — we keep the restriction bounded so the job terminates and the mechanics stay clear.

Beam APIs introduced

  • apache_beam.transforms.core.RestrictionProvider — tells Beam how to make, split, and size a restriction for an element (this is what makes a DoFn splittable).
  • apache_beam.io.restriction_trackers.OffsetRange — the restriction type: a half-open interval [start, stop) with .split() and .size().
  • apache_beam.io.restriction_trackers.OffsetRangeTracker — the matching tracker, implementing try_claim() and try_split().
  • beam.DoFn.RestrictionParam(provider) — the process() parameter whose presence upgrades the DoFn to splittable and injects a live tracker.

The example pipeline

A toy bounded SDF that "reads" the integer range [0, N) and emits each offset. The provider models the work; the process() claim loop is the heart of every SDF: 

class RangeRestrictionProvider(RestrictionProvider):
    def initial_restriction(self, element):   # full work for element N
        return OffsetRange(0, element)        #   -> [0, N)
    def create_tracker(self, restriction):    # stateful claim-gate
        return OffsetRangeTracker(restriction)
    def restriction_size(self, element, restriction):
        return restriction.size()             # weigh splits by # positions
    def split(self, element, restriction):    # OPTIONAL initial fan-out
        yield from restriction.split(desired_num_offsets_per_split=8,
                                     min_num_offsets_per_split=1)

class CountingSource(beam.DoFn):
    def process(self, element,
                # presence of RestrictionParam = this DoFn is splittable
                restriction_tracker=beam.DoFn.RestrictionParam(RangeRestrictionProvider())):
        cur = restriction_tracker.current_restriction()   # the sub-range we own
        pos = cur.start
        while restriction_tracker.try_claim(pos):  # True=ours, False=split away -> stop
            yield pos                              # "read" position pos
            pos += 1

Wiring it up is a normal ParDo — Beam recognises the RestrictionParam and runs the SDF expansion (pair-with-restriction → split-and-size → process-sized-elements): 

offsets = (
    p
    | "TheNumberN"      >> beam.Create([known.count])   # one element: N
    | "ReadRangeViaSDF" >> beam.ParDo(CountingSource())  # 1 element -> up to N claims
)
