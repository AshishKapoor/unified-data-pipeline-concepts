Ch 05 — Keyed Aggregation: GroupByKey, CoGroupByKey, Flatten

Three keyed/collection primitives underpin everything else: GroupByKey (the shuffle), CoGroupByKey (relational joins), and Flatten (union). Master the shuffle and you understand why pipelines cost what they cost.

Concept

Most useful aggregations are keyed: "count clicks per user", "join events to a profile by user_id". In Beam a keyed element is just a KV — a 2-tuple (key, value) — and the keyed transforms key off element[0].

GroupByKey (GBK) is the fundamental keyed primitive. Given a PCollection of (K, V) it returns (K, Iterable[V]): it brings every value for a key onto the same worker. That movement of data across the network is the shuffle, and it is simultaneously the most important and the most expensive thing a runner does. On Flink, a GBK becomes a keyed network exchange — records are hash-partitioned by key and redistributed so the runner can gather a key's values together.

🎯
The one idea for this chapter: grouping is not free. GBK shuffles all your data over the network. Everything you learn later about Combine (Ch 6), windows, and triggers is, in part, about controlling what gets shuffled and when.

CoGroupByKey (CoGBK) is GBK generalised to several inputs — a relational join. You hand it a dict of tagged keyed collections; it co-groups them by their common key and returns, per key, the matching values from each side. We use it to enrich every click with the user's profile. If one side has no match (user u4 has clicks but no profile), that side simply comes back empty — giving you a left/outer-join shape that you handle yourself.

Flatten is the odd one out: it does not group or shuffle. It unions N collections of the same element type into one — think SQL UNION ALL. We use it to merge two click streams (e.g. two regions) into a single collection before grouping.

⚠️
On unbounded data, GBK/CoGBK require a window. You cannot group "all values for a key" if the key never stops receiving values. A window bounds the grouping in event time. This chapter is bounded, so the default global window is implicit and we don't declare one — but Ch 9 onward you must.

Beam APIs introduced

  • beam.GroupByKey — (K, V) → (K, Iterable[V]); the shuffle primitive.
  • beam.CoGroupByKey — co-group several tagged keyed PCollections; a relational join.
  • beam.Flatten — union N PCollections of the same type (no grouping, no shuffle).
  • KV — a plain (key, value) tuple; keyed transforms key off element[0].
  • beam.FlatMap / beam.Map — used here to format and log the grouped/joined rows.

The example pipeline

Two keyed collections — clicks and profiles, both keyed by user_id. We Flatten two click streams, GroupByKey the union, and CoGroupByKey clicks against profiles to enrich each click: 

with beam.Pipeline(options=options) as p:
    clicks      = p | "Clicks"     >> beam.Create(CLICKS)       # (user_id, path)
    more_clicks = p | "MoreClicks" >> beam.Create(MORE_CLICKS)  # a second click stream
    profiles    = p | "Profiles"   >> beam.Create(PROFILES)     # (user_id, {name, plan})

    # Flatten: UNION two click streams (same element type) — no shuffle.
    all_clicks = (clicks, more_clicks) | "UnionClicks" >> beam.Flatten()

    # GroupByKey: the shuffle. (user_id, path) -> (user_id, [paths...]).
    (all_clicks
        | "GroupClicksByUser" >> beam.GroupByKey()
        | "LogGrouped"        >> beam.Map(lambda kv: logging.info(format_grouped(kv))))

    # CoGroupByKey: relational join, enrich each click with its profile.
    ({"clicks": all_clicks, "profile": profiles}
        | "JoinClicksToProfiles" >> beam.CoGroupByKey()
        | "Enrich"               >> beam.FlatMap(enrich_with_profile)
        | "LogEnriched"          >> beam.Map(lambda line: logging.info(line)))
