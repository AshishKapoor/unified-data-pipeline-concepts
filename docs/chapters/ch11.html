Ch 11 — Triggers & Accumulation Modes

A window says which events group together; the watermark says when they're probably all in. The trigger is the third leg: it decides WHEN a window emits — and a window can emit many times, producing a sequence of panes. The accumulation mode decides how those panes relate: each one a running total, or each one only the new delta.

Concept

In Ch 9 we put events into windows; in Ch 10 the watermark estimated event-time completeness. But a window still needs a rule for pulling the trigger — actually materializing a result. That rule is the trigger. Crucially, a window is not limited to one output: a single window can fire repeatedly, each firing emitting a pane.

The default and most useful skeleton is AfterWatermark, which lets you attach three kinds of firing:

  • early (early=) — speculative panes that fire before the watermark reaches the window's end, so a dashboard isn't blank while data trickles in. Driven by AfterProcessingTime(n) (every n seconds of wall clock) or AfterCount(n) (every n elements).
  • on-time — exactly one pane, fired when the watermark crosses the end of the window. This is the "the data is probably all here now" result.
  • late (late=) — panes fired after the on-time pane, for elements that arrive late but still within allowed_lateness (Ch 12). AfterCount(1) means "re-fire on every late element".
🎯
The one idea for this chapter: the trigger answers WHEN, and the accumulation mode answers HOW the panes combine. Change the mode alone — same window, same trigger, same data — and every pane value changes.

Accumulation: ACCUMULATING vs DISCARDING

When a window fires more than once, what does each pane contain? That's the accumulation_mode:

  • ACCUMULATING — each pane re-emits the running total including all data seen so far. Successive panes grow monotonically: 3 → 7 → 12 → …. Use when the sink overwrites by key (an upsert / last-write-wins dashboard).
  • DISCARDING — the runner clears the window's accumulated state after each firing, so each pane carries only the delta since the last pane: 3 → 4 → 5 → …. Use when the sink adds (you sum the panes downstream).
⚠️
Double-counting trap: pick the mode that matches your sink. ACCUMULATING panes into an additive sink double-counts everything; DISCARDING panes into an overwriting sink throws away earlier deltas. The numbers in the right-hand animation show exactly how the two diverge from the same firings.

Composing triggers

Beyond AfterWatermark you can build richer rules: Repeatedly (run a sub-trigger forever), AfterAny / AfterAll (fire when any / all of several sub-triggers are ready), and AfterEach (advance through sub-triggers in sequence). AfterWatermark(early=…, late=…) is itself sugar over these.

Beam APIs introduced

  • AfterWatermark(early=…, late=…) — the early / on-time / late firing skeleton.
  • AfterProcessingTime(n) — fire every n seconds of processing (wall-clock) time.
  • AfterCount(n) — fire after n elements have accumulated.
  • Repeatedly, AfterAny, AfterAll, AfterEach — trigger combinators.
  • AccumulationMode.ACCUMULATING / DISCARDING — how successive panes relate.
  • beam.DoFn.PaneInfoParam — per-pane metadata (timing, index, is_first/is_last).

The example pipeline

One windowed sum, run twice with the same trigger but different accumulation modes, so you can read cumulative-vs-delta side by side. The streaming source injects event-time lag so the watermark trails and early panes get a chance to fire: 

trigger = AfterWatermark(
    early=AfterProcessingTime(10),  # speculative early panes every 10s
    late=AfterCount(1),             # one extra pane per late element
)

def windowed_sum(events, *, accumulation_mode, mode_label):
    return (
        events
        | f"ToKV-{mode_label}"  >> beam.Map(lambda e: (e["key"], e["value"]))
        | f"Window-{mode_label}" >> beam.WindowInto(
            FixedWindows(30),
            trigger=trigger,
            accumulation_mode=accumulation_mode,   # the ONLY thing that differs
            allowed_lateness=60,
        )
        | f"SumPerKey-{mode_label}" >> beam.CombinePerKey(sum)
        | f"LogPane-{mode_label}"   >> beam.ParDo(LogPane(mode_label))  # logs PaneInfo
    )

windowed_sum(events, accumulation_mode=AccumulationMode.ACCUMULATING, mode_label="ACCUMULATING")
windowed_sum(events, accumulation_mode=AccumulationMode.DISCARDING,  mode_label="DISCARDING")

LogPane reads DoFn.PaneInfoParam and prints the firing class — EARLY / ON_TIME / LATE — plus the pane index and whether it is the first/last pane. Watch the two branches in the Flink TaskManager logs and compare the value streams: ACCUMULATING climbs; DISCARDING resets after every pane.
