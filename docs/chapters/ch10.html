Ch 10 — Watermarks: How the System Knows Event Time Advanced

A watermark W(t) is the runner's promise: "I have now seen every element with event time ≤ W." It is the heartbeat that closes windows — and it propagates as the minimum across an operator's input channels.

Concept

Windows (Ch 9) chop the timeline into buckets, but a bucket is useless until the system can say "this bucket is complete — no more data is coming for it." In a stream there is no end-of-file to signal that. The mechanism that fills the gap is the watermark: a monotonically advancing event-time value that travels through the pipeline alongside the data. At processing-time t, a watermark of W(t) asserts that the runner believes all elements with event time ≤ W(t) have already arrived. It is, in one phrase, the system's estimate of completeness for the past.

Crucially, watermarks lag the wall clock. If events arrive a few seconds after the moment they describe (network hops, batching, retries), the watermark trails processing time by roughly that amount. We make this visible in the lab by stamping every event a fixed lag_sec in the past, so the source watermark always sits behind "now".

🎯
The one idea for this chapter: a window does not fire when wall-clock time says it should — it fires when the watermark crosses the window's end. Watermark advance is literally what closes windows.

Perfect vs heuristic watermarks

A perfect watermark is one the source can prove: replaying a file in event-time order, or reading a partition whose timestamps are monotonic, lets the source assert completeness with certainty. Most real sources can't prove it and instead emit a heuristic watermark — a best guess based on observed skew. The trade-off is sharp: a heuristic that runs ahead of reality declares windows complete too early and creates late data (Ch 12); one that lags too far conservatively wastes latency, holding results back longer than necessary.

Propagation: MIN across input channels

An operator rarely has a single input. A GroupByKey is fed by a shuffle from every upstream partition; a flatten merges several streams. Such an operator can only claim completeness up to its slowest input — so its output watermark is the min over all input channels. This is the rule that gives watermarks their teeth: a single slow input drags the whole downstream min down with it.

⚠️
The idle-input trap. If one input channel goes silent — not finished, just temporarily producing nothing — its watermark stops advancing, and because the output is a min, the entire downstream watermark freezes. Windows stop firing even though data is flowing on the other channels. Flink's withIdleness fixes this by marking a quiet source idle so it stops "voting" in the min, letting the live channels advance the watermark again.

How Flink and Beam cooperate

On the portable runner the division of labour is clean. Flink owns watermark generation and propagation: each operator subtask carries a "Low Watermark" you can read in the Web UI, computed by exactly the min rule above. Beam consumes that watermark to decide pane firing — with the default AfterWatermark trigger (Ch 11), a window emits the instant the watermark passes its end, and its per-key state is then garbage-collected (subject to allowed lateness, Ch 12).

Beam APIs introduced

  • Watermarks — the runner-maintained event-time frontier; on Flink, the per-operator "Low Watermark" surfaced in the Web UI.
  • beam.WindowInto(FixedWindows(n)) — assigns elements to fixed windows whose ends the watermark must cross to fire.
  • DoFn.WindowParam / DoFn.PaneInfoParam — inspect the firing window's bounds and pane timing from inside a DoFn.
  • SyntheticEvents(lag_sec=…) — our shared source that stamps events in the past so the watermark visibly trails processing time.
  • Default AfterWatermark trigger — the link between watermark advance and a window firing (detailed in Ch 11).

The example pipeline

An unbounded stream whose event times trail wall-clock by lag_sec, windowed into fixed buckets. The LogFiring DoFn logs each firing relative to the watermark, so you can see windows close ~lag_sec seconds after their wall-clock end: 

with beam.Pipeline(options=options) as p:
    (
        p
        # event_time = processing_time - lag  → source watermark trails the wall clock
        | "SyntheticEvents" >> SyntheticEvents(events_per_sec=4, lag_sec=known.event_time_lag)
        | "KeyBySensor"     >> beam.Map(lambda e: (e["key"], e["value"]))
        | "FixedWindows"    >> beam.WindowInto(FixedWindows(known.window_size))
        | "CountPerKeyWindow" >> combiners.Count.PerKey()   # fires when watermark > window end
        | "LogFiring"       >> beam.ParDo(LogWindowFiring())  # logs firing vs the watermark
    )

The LogWindowFiring DoFn reads DoFn.WindowParam to get the window's [start, end) and reports how far window.end trails the current wall clock — a direct readout of how long the watermark took to cross it.
