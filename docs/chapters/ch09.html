Ch 09 — Windowing: Fixed, Sliding, Sessions, Global

A window is a per-element tag. WindowInto runs each element's event timestamp through a WindowFn that assigns it to one or more windows — and grouping is always per key, per window.

Concept

Aggregating an unbounded stream is impossible without slicing time into finite chunks — you can never wait for "all" the data. Windowing is that slicing. Crucially, it is not itself an aggregation: beam.WindowInto(fn) simply decorates every element with the window(s) it belongs to, by running the element's event timestamp (assigned upstream, Ch 8) through a WindowFn. The grouping happens later, in a GroupByKey/CombinePerKey, which produces exactly one output per (key, window) pair.

Because windowing keys off event time, arrival order is irrelevant: a deliberately shuffled, out-of-order stream produces identical windows. That is why this chapter replays the same event log through four different WindowFns and only the window assignment changes.

The four window functions

  • FixedWindows(60) — tumbling: contiguous, non-overlapping 60s buckets aligned to the epoch ([0,60), [60,120), …). Each element lands in exactly one window.
  • SlidingWindows(60, 30) — overlapping: each window is 60s wide and a new one starts every 30s, so every element lands in two windows (size / period = 2). Use it for rolling/moving aggregates ("last 60s, updated every 30s").
  • Sessions(gap_size=45) — data-driven and merging: each element seeds a window [t, t+gap); windows that touch or overlap are merged during grouping. A burst of activity spaced under 45s collapses into one session; a quiet stretch of ≥ 45s starts a fresh one. The window bounds depend on the data, not the clock.
  • GlobalWindows() — a single window spanning all of time. It is the default for batch; for unbounded data it never closes, so it is useless for aggregation unless paired with a trigger (Ch 11).
🎯
The one idea for this chapter: windowing is an orthogonal knob. The pipeline shape — window → pair-with-one → count-per-key → log — is byte-for-byte identical across all four; only the WindowFn argument to WindowInto differs.
⚠️
Merging is per key. Sessions merge windows independently for each key — sensor-0 and sensor-1 can have completely different session boundaries from the very same stream. Merging WindowFns also can't be used as a GroupByKey input more than once without re-windowing.

Beam APIs introduced

  • beam.WindowInto(window_fn) — assigns each element to its window(s).
  • window.FixedWindows(size) — tumbling, non-overlapping buckets.
  • window.SlidingWindows(size, period) — overlapping rolling windows.
  • window.Sessions(gap_size) — data-driven, merging windows.
  • window.GlobalWindows() — the single all-of-time window (batch default).
  • DoFn.WindowParam — injects the current window so you can log/inspect it.

The example pipeline

One replayed, out-of-order stream; the same four-step branch applied with three different WindowFns. DoFn.WindowParam logs which window each per-key count fell in: 

def count_per_key(events, window_fn, label):
    return (
        events
        | f"Window/{label}" >> beam.WindowInto(window_fn)      # tag each element with its window(s)
        | f"Pair/{label}"   >> beam.Map(lambda e: (e["key"], 1))
        | f"Count/{label}"  >> combiners.Count.PerKey()        # one (key,count) per (key, window)
        | f"Log/{label}"    >> beam.ParDo(LogPerKeyWindow(label))
    )

with beam.Pipeline(options=options) as p:                       # streaming=True
    events = p | "Replay" >> ReplayEvents(RAW_EVENTS, emission_order=EMISSION_ORDER)
    count_per_key(events, FixedWindows(60),            "FIXED")
    count_per_key(events, SlidingWindows(size=60, period=30), "SLIDING")
    count_per_key(events, Sessions(gap_size=45),       "SESSION")
    count_per_key(events, GlobalWindows(),             "GLOBAL")

LogPerKeyWindow reads window.start/window.end via DoFn.WindowParam and logs lines like [FIXED] window=[60,120) key=sensor-0 count=3. Compare the [FIXED]/[SLIDING]/[SESSION]/[GLOBAL] lines for the same key to see the assignment difference directly.
