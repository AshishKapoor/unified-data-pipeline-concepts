Ch 16 — Exactly-Once, Fault Tolerance & the Flink Runtime

Barriers flow inline with your records, align at multi-input operators, and trigger a consistent snapshot — that is how Flink gives you exactly-once state. Getting exactly-once output is a separate problem solved by transactional sinks.

Concept — Asynchronous Barrier Snapshotting

Flink's fault-tolerance is built on Asynchronous Barrier Snapshotting (ABS), a streaming variant of the classic Chandy–Lamport distributed-snapshot algorithm. The JobManager's checkpoint coordinator periodically injects a numbered barrier into every source. The barrier is not a side channel — it travels inline, in order, mixed with the actual records.

When an operator with more than one input receives a barrier on one input, it aligns: it buffers that input and waits until the same barrier arrives on every other input. Alignment guarantees the snapshot reflects a consistent cut of the dataflow — every operator's state corresponds to "all records before barrier n, none after." Once aligned, the operator snapshots its state asynchronously to the state backend and forwards the barrier downstream. When all operators and sinks acknowledge barrier n, checkpoint n is complete and durable.

🎯
The one idea for this chapter: a checkpoint is a globally consistent cut taken without stopping the stream. Combined with a Beam bundle as the unit of commit, re-processing after a crash replays only the records since the last completed checkpoint — so counted state is restored, never double-counted.

Exactly-once STATE is not exactly-once OUTPUT

On recovery Flink rewinds the sources and replays. Replay means a record can be re-emitted to your sink. Checkpointing alone therefore gives exactly-once state but only at-least-once at the boundary. End-to-end exactly-once additionally needs the sink to be:

  • Transactional — Kafka EOS via a transactional producer (two-phase commit): a checkpoint's output is committed atomically and a read_committed consumer never sees the records of an aborted attempt.
  • Idempotent — a keyed upsert sink where re-writing the same key is a no-op, achieving the same effect without transactions.
⚠️
Do not conflate the two. "Flink has exactly-once" refers to state. Whether your downstream system sees each result once depends entirely on the sink's commit protocol.

Savepoints, rescaling & the runtime

Checkpoints are automatic, periodic, runtime-owned, and (by default) cleaned up. Savepoints are manual, durable, portable images you trigger for upgrades and rescaling. To rescale, take a savepoint, then restore at a new parallelism; Flink redistributes keyed state across the new task slots via key-group reassignment: 

flink savepoint <jobId> s3://bucket/savepoints
flink run -s s3://bucket/savepoints/savepoint-xxxx -p 4 job.jar   # restore at parallelism 4

The runtime splits into a JobManager (coordinates checkpoints, schedules the JobGraph) and TaskManagers (run the operators in task slots; total slots = available parallelism). Between operators, credit-based flow control propagates backpressure: a slow downstream operator grants fewer buffer credits, throttling upstream rather than dropping data. Watch all of this over the Flink REST API: /jobs/:id/checkpoints and the back-pressure endpoint. The state backend is configurable — hashmap (on-heap, fast, bounded by RAM) vs rocksdb (on-disk, supports very large state).

Beam APIs introduced

  • execution.checkpointing.interval — how often barriers are injected (set on the cluster via FLINK_PROPERTIES to 10s; smaller = less replay, more overhead).
  • portable_options(job_name, streaming=True) — unbounded job, so checkpointing is active.
  • ReadFromKafka — a replayable source; offsets are committed with the checkpoint (enable.auto.commit=false).
  • WriteToKafka — the sink; configure a transactional producer for exactly-once output.
  • Flink REST /jobs/:id/checkpoints + state.backend (hashmap / rocksdb) + savepoint for rescaling.

The example pipeline

A Kafka streaming job (read clicks → window → count per user → write back to Kafka). The transform graph is ordinary; the point is what survives a crash. The per-key window counts are the state that ABS snapshots at each barrier: 

with beam.Pipeline(options=options) as p:   # portable_options(job_name, streaming=True)
    counts = (
        p
        | "ReadClicks"   >> ReadFromKafka(           # replayable source; offsets ride the checkpoint
              consumer_config={"bootstrap.servers": bootstrap,
                               "enable.auto.commit": "false"},  # offsets owned by the checkpoint
              topics=[known.input_topic])
        | "KeyByUser"    >> beam.Map(lambda kv: (kv[0].decode() if kv[0] else "anon", 1))
        | "Window"       >> beam.WindowInto(beam.window.FixedWindows(known.window_sec))
        | "CountPerUser" >> beam.CombinePerKey(sum)   # this per-key count is the snapshotted STATE
        | "ToKafkaRecord">> beam.MapTuple(to_count_record)
    )
    # For exactly-once OUTPUT, configure a transactional (EOS) producer on this sink:
    counts | "WriteCounts" >> WriteToKafka(producer_config={"bootstrap.servers": bootstrap},
                                           topic=known.output_topic)

Kill a TaskManager — docker compose kill taskmanager — and Flink rolls every operator back to the last completed checkpoint and replays Kafka from the offsets stored in it. Because Kafka is replayable and the counts were snapshotted, the per-window counts come out identical: no double count. To rescale, take a savepoint and resubmit with a higher --parallelism.
