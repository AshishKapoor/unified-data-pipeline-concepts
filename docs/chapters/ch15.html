Ch 15 — IO Connectors & Cross-Language: Files and KafkaIO

Files are bounded IO; Kafka is unbounded IO. And Python gets the production KafkaIO by borrowing a Java transform through the cross-language expansion service.

Concept: bounded vs unbounded IO

Every Beam pipeline starts at a source. Sources come in two flavours, and the flavour decides everything downstream:

  • Bounded IO — a source with a known end. beam.io.ReadFromText (from Ch 1) reads every line of a file set, emits them, then declares itself complete. The watermark advances to +∞ and the job terminates. Batch.
  • Unbounded IO — a source that never declares completion. A Kafka topic is an endless append-only log, so KafkaIO is a streaming source: it advances a watermark forever and the job keeps running until you cancel it. This is why this chapter is streaming=True.

TextIO is pure Python and "just works". KafkaIO is the interesting case, and the reason is not about Kafka at all — it is about languages.

Concept: KafkaIO is a Java transform

There is no production-grade native-Python KafkaIO. The mature, exactly-once KafkaIO lives in the Java SDK. Rather than reimplement it, Beam lets the Python SDK reuse the Java transform through its cross-language (xlang) framework. The Python ReadFromKafka / WriteToKafka you import are thin stubs that drive a Java expansion service behind the scenes.

🎯
The one idea for this chapter: a transform can be implemented in a different language than the pipeline that uses it. At construction time Python asks a Java expansion service to expand KafkaIO; at runtime a Java harness and a Python harness run side by side, exchanging records over the Fn API.

Construction time: the expansion handshake

When you call ReadFromKafka(...), the Python SDK opens a gRPC connection to the expansion service — by default the one bundled in the Beam Flink job server on port 8097. It sends the transform's URN and parameters; the service expands KafkaIO into a real Java sub-graph and returns it as a proto. Python splices that Java sub-graph into the pipeline it is building. Your single Python file now contains edges that are defined in Java.

Runtime: two harnesses, one job

At execution, the portable runner starts two SDK harnesses for this one job: a Java harness that runs the Kafka read/write, and a Python harness that runs your CombinePerKey and Map DoFns. Records cross the language boundary over the Fn API (a gRPC data plane). You wrote Python, but the Kafka edges execute as Java.

⚠️
What "Run on Flink" actually runs here. Running the Java KafkaIO harness on a portable Flink cluster needs a Java SDK worker pool (the Java analogue of this project's Python pool) — genuinely involved infra on a Flink EXTERNAL-worker setup. So by default the Run button runs a pure-Python streaming demo (an unbounded synthetic click stream, windowed and counted per user) that has the same streaming + windowed-aggregation shape and runs flawlessly out of the box.

The real cross-language KafkaIO code is exactly what you see above (build_kafka_pipeline) — that is the lesson. To execute it: 
./scripts/up-kafka.sh && ./scripts/seed-kafka.sh   # broker + topics
# provide a Java SDK worker pool at $BEAM_JAVA_WORKER_POOL (advanced — see README), then:
ENABLE_XLANG_KAFKA=1 ./scripts/submit.sh ch15
Brokers must be reached at kafka:29092 (in-network), not localhost:9092 — SDK harnesses run inside the cluster network.
🧱
Bytes in, bytes out. ReadFromKafka yields (key_bytes, value_bytes) and WriteToKafka expects (bytes, bytes) — it does not serialize Python objects for you. Decode on the way in, encode on the way out.

Beam APIs introduced

  • beam.io.ReadFromText / WriteToText — bounded file IO (Ch 1 recap).
  • apache_beam.io.kafka.ReadFromKafka — unbounded Kafka source (Java, via xlang).
  • apache_beam.io.kafka.WriteToKafka — Kafka sink (Java, via xlang).
  • expansion_service= — the host:port of the Java expansion service (:8097).
  • consumer_config / producer_config — passed straight to the Kafka client; set bootstrap.servers here.
  • beam.window.FixedWindows + CombinePerKey — windowed aggregation over an unbounded stream.

The example pipeline

Read clicks from clicks-in, count clicks per user in 10-second windows, and write the counts back to counts-out. The Kafka edges are Java; the middle is Python: 

with beam.Pipeline(options=options) as p:        # streaming=True
    counts = (
        p
        # JAVA transform, expanded via the cross-language service on :8097
        | "ReadFromKafka" >> ReadFromKafka(
            consumer_config={
                "bootstrap.servers": kafka_bootstrap(),  # kafka:29092 (in-network!)
                "auto.offset.reset": "earliest",
            },
            topics=["clicks-in"],
            expansion_service="localhost:8097",
        )
        # --- the PYTHON harness runs everything below ---
        | "ToUserOne"   >> beam.Map(to_user_one)             # (k,v) bytes -> (user, 1)
        | "Window"      >> beam.WindowInto(beam.window.FixedWindows(10))
        | "CountPerUser">> beam.CombinePerKey(sum)           # clicks per user per window
        | "Encode"      >> beam.Map(encode_result)           # (user, n) -> (bytes, bytes)
    )
    # JAVA transform again: write per-window counts back to Kafka
    counts | "WriteToKafka" >> WriteToKafka(
        producer_config={"bootstrap.servers": kafka_bootstrap()},
        topic="counts-out",
        expansion_service="localhost:8097",
    )

The submitter ships this mixed-language graph to the Beam Flink Job Server. The expansion of KafkaIO happens during construction; at runtime the Flink JobManager schedules both the Java and Python harnesses. Open the Flink UI to watch the (never-ending) streaming job.

idle Open in Flink UI ↗

The log streams the submitter's output live over SSE. This is an unbounded job — it runs until you hit Cancel. Per-user counts land on the counts-out topic; consume it with kafka-console-consumer --topic counts-out --from-beginning --property print.key=true.
